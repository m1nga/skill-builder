# Diagnostic Lenses

Use these lenses selectively. They are questions for locating the limiting condition, not stages a
project must complete.

## Contents

1. Evidence survival
2. Authority coherence
3. Purpose clarity
4. Reality contact
5. Implementation leverage
6. Context contamination
7. Change and reversibility
8. Route signals

## 1. Evidence survival

- What original material still exists?
- Which artifacts are exact, partial, reconstructed, summarized, or remembered?
- Is there a remote repository, backup, snapshot, export, session log, attachment, cache, or
  deployed copy?
- Would further writing overwrite or reduce recovery options?
- Can each recovered artifact retain provenance?

If exact evidence may still be recoverable, choose **Recover** before product redesign.

## 2. Authority coherence

- Is there exactly one active answer to the consequential questions?
- Which statements came from the user, customers, observed results, code, or an AI?
- Are proposals labeled as decisions?
- Do active files disagree?
- Does a later correction replace earlier active truth, or merely sit beside it?
- Can a zero-context reader identify the current authority without reading an archive?

If purpose is stable but authority is scattered, choose **Repair**. If authority is inseparable
from a replaced direction, choose **Rebuild**.

## 3. Purpose clarity

- Who experiences the problem?
- What change should become true for them?
- Why does this need to exist instead of an available alternative?
- What durable mandate should constrain the product?
- What is explicitly outside the product?
- Which parts are user decisions, current hypotheses, and unresolved questions?

If these answers cannot govern trade-offs, choose **Clarify** even when substantial code exists.

## 4. Reality contact

- Which claims have been tested with real users, data, publishing, payment, operation, or external
  constraints?
- Which tests prove internal mechanisms only?
- Has the project repeatedly replaced external contact with more construction?
- What is the cheapest real probe that could disprove the current direction?

If the foundation is coherent but value remains untested, choose **Continue** with a vertical probe
rather than rebuild.

## 5. Implementation leverage

- Which mechanisms have behavioral evidence or meaningful tests?
- Which code only embodies a superseded strategy?
- Can a useful mechanism be re-derived cheaply from the new purpose?
- Would migration cost more than reimplementation?
- Does reuse import vocabulary, data shapes, or assumptions that would distort the new direction?

Keep implementation only when current requirements independently demand it. Prior effort is not a
requirement.

## 6. Context contamination

- Do active summaries preserve dead users, pricing, channels, identities, architecture, or product
  categories?
- Does a clean session automatically load old instructions, skills, histories, or parent-directory
  files?
- Does an archive map invite agents to retrieve obsolete material?
- Is the session writing the clean foundation already saturated with old framing?

Use a separated source bundle and fresh builder when contamination is structural. Physical deletion
is not the first requirement for clean reasoning.

## 7. Change and reversibility

- Is the proposed action reversible?
- Is the exact target independently understood?
- Is it the only surviving source for another product or purpose?
- Has a recovery path been verified rather than assumed?
- Can a shadow copy, worktree, branch, export, or new directory test the intervention first?

No amount of confidence substitutes for target-specific authority on irreversible actions.

## 8. Route signals

| Evidence pattern | Primary route | First useful move |
|---|---|---|
| Important sources missing; backups/logs may survive | Recover | Freeze writes and inventory recovery sources |
| Many artifacts; no stable purpose or user outcome | Clarify | Restore the purpose and decision boundary |
| Purpose stable; active docs/code contradict it | Repair | Map affected dependencies and replace stale truth |
| Direction replaced; old context repeatedly returns | Rebuild | Build a clean authorized source bundle |
| Foundation coherent; real value or operation untested | Continue | Run one end-to-end external probe |
| Low evidence and no current value case | Hold / retire | Preserve provenance and stop construction |

Mixed cases are normal. Name the route that removes the current bottleneck; reassess afterward.
